;; Fermat's Little Theorem: if n is prime, then for any a < n:
;;   a^n ≡ a (mod n)
;; Or equivalently: a^(n-1) ≡ 1 (mod n)

(define (square x) (* x x))

;; Fast modular exponentiation: computes (base^exp) mod m
;; Key insight: (a * b) mod m = ((a mod m) * (b mod m)) mod m
;; So we can keep numbers small by taking mod at each step
(define (expmod base exp m)
  (cond ((= exp 0) 1)
        ((even? exp)
         (remainder (square (expmod base (/ exp 2) m))
                    m))
        (else
         (remainder (* base (expmod base (- exp 1) m))
                    m))))

;; Fermat test: pick random a, check if a^n ≡ a (mod n)
(define (fermat-test n)
  (define (try-it a)
    (= (expmod a n n) a))
  (try-it (+ 1 (random (- n 1)))))

;; Run the test 'times' times for confidence
;; One failure = definitely not prime
;; All pass = probably prime (higher 'times' = more confident)
(define (fast-prime? n times)
  (cond ((= times 0) true)
        ((fermat-test n) (fast-prime? n (- times 1)))
        (else false)))

;; Modified timed-prime-test using fast-prime?
(define (timed-prime-test n)
  (newline)
  (display n)
  (start-prime-test n (runtime)))

**The logic:**

1. **`expmod`** - Computes `base^exp mod m` efficiently using successive squaring. Instead of computing the huge number `base^exp` then taking mod, we take mod at each step to keep numbers small. This is Θ(log n) since we halve the exponent each time.

2. **`fermat-test`** - Picks a random `a` between 1 and n-1, checks if `a^n ≡ a (mod n)`. If true, n *might* be prime. If false, n is *definitely* composite.

3. **`fast-prime?`** - Runs the Fermat test multiple times. More iterations = higher confidence. We use 10 iterations.

**Expected timing analysis:**

Since Fermat test is Θ(log n):
- Testing 1000: proportional to log(1000) ≈ 10
- Testing 1,000,000: proportional to log(1,000,000) ≈ 20

So primes near 1,000,000 should take only about **2× longer** than primes near 1000, not 31× longer like with the √n algorithm!

**Possible discrepancies:**

1. **Numbers are still small** - On modern machines, all these run in microseconds, hard to measure
2. **`random` overhead** - The random number generation adds constant overhead that may dominate for small inputs
3. **Arithmetic on larger numbers** - Even though it's log(n) steps, each `expmod` step on larger numbers involves larger intermediate values, which takes slightly more time
